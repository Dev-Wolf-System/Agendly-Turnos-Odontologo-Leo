  Propuestas Estratégicas

  1. Portal de Registro y Onboarding (Landing Page)

  Propongo crear una landing page integrada en el mismo proyecto Next.js (ruta / y /planes) en lugar de un sitio aparte.  
  Razones:
  - Un solo deploy, un solo dominio
  - SEO directo hacia la app
  - Conversión inmediata (ver plan → registrarse → entrar)

  Flujo propuesto:
  Landing (/) → Planes (/planes) → Registro (/registro?plan=xxx) → Wizard Config → Dashboard

  Landing page:
  - Hero con gradiente premium, screenshots del sistema
  - Sección "Para quién" (clínicas, consultorios, centros médicos)
  - Features con iconos animados
  - Testimonios / métricas
  - CTA → ir a planes

  Página de Planes (/planes):
  - Cards comparativas lado a lado (como Stripe pricing)
  - Toggle mensual/anual con descuento
  - Botón "Empezar trial gratuito" por plan
  - Feature comparison table debajo

  Registro (/registro?plan=xxx):
  - Wizard de 3 pasos: (1) Datos clínica, (2) Datos admin, (3) Selección especialidad + horarios
  - Al terminar → crea clínica + usuario + suscripción trial del plan elegido
  - Redirige directo al dashboard con onboarding tour

  2. Documentos/Imágenes en Historial Médico

  Propongo:
  - Nueva entidad ArchivoMedico (paciente_id, nombre, tipo, url, size, uploaded_by)
  - Almacenamiento: S3/MinIO para producción, o disco local inicialmente
  - En la ficha del paciente: tab "Documentos" con upload drag-and-drop
  - Soportar: imágenes (radiografías, fotos), PDFs (estudios, recetas), documentos
  - Preview inline para imágenes, descarga para PDFs

  3. Horarios por Profesional

  Propongo:
  - Nueva entidad HorarioProfesional (user_id, clinica_id, horarios jsonb)
  - Misma estructura de horarios que la clínica (lunes-domingo, mañana/tarde)
  - En configuración → tab Equipo: cada profesional puede tener su propio horario
  - El cálculo de turnos disponibles usa el horario del profesional si existe, si no el de la clínica
  - El agente IA usa estos horarios al mostrar disponibilidad

  4. Reportes Avanzados

  Propongo estos reportes:
  - Productividad por profesional: turnos completados, tasa de cancelación, ingresos generados
  - Análisis de pacientes: nuevos vs recurrentes, frecuencia de visitas, retención
  - Financiero: ingresos por período, por tratamiento, por método de pago, proyecciones
  - Ocupación: % de slots ocupados por día/semana, horarios más demandados
  - Tratamientos: más realizados, ingresos por tipo, duración promedio real vs estimada
  - Export: PDF con branding de la clínica + Excel descargable

  5. Agente IA (Zoe) — Arquitectura

  Propongo: Un solo agente multi-tenant (no uno por clínica). Razones:
  - Menor costo operativo (1 flujo n8n, no N)
  - Actualizaciones centralizadas
  - La API ya es multi-tenant (cada request lleva x-api-key + clinica_id de la instancia Evolution)

  Flujo:
  WhatsApp (paciente) → Evolution API → Webhook n8n → Agente Zoe → API Avax Health → Respuesta → Evolution → WhatsApp     

  Capacidades del paciente vía WhatsApp:
  - Ver turnos disponibles (por especialidad/profesional)
  - Registrarse si no existe
  - Agendar turno
  - Cancelar/reprogramar turno
  - Consultar info de la clínica (dirección, horarios, tratamientos, precios)
  - Recibir recordatorios automáticos

  Capacidades del admin vía WhatsApp:
  - Consultar turnos del día/semana
  - Ver resumen financiero
  - Estado de la agenda (% ocupación)
  - Alertas de stock bajo
  - Notificaciones de nuevos turnos/cancelaciones

  Arquitectura n8n:
  - 1 workflow principal con router por clinica_id (extraído del webhook)
  - Sub-workflows reutilizables (buscar paciente, crear turno, consultar horarios)
  - Contexto de conversación en Redis (para mantener el hilo del chat)
  - OpenAI como cerebro para interpretar mensajes y decidir acciones
